How to use this

This is the answer key for the upgrade gauntlet. Try the drill first - stand the cluster up, seed the gotchas (make seed), point kubectl at it, and hunt. Then come back and grade yourself.

Each item below is written as Who / What / Where / When / Why, then a Detect command and a Fix. The upgrade path is 1.34 → 1.35 → 1.36, one minor at a time.

🔻 When you're done: make down ENV=dev. A left-running control plane is ~$73/mo; on an extended-support version it is ~$438/mo.

0 · Golden rules of an EKS upgrade

  1. Never skip a minor. EKS upgrades one minor at a time. 1.34 → 1.36 means doing 1.34→1.35, then 1.35→1.36. Trying to jump is rejected.
  2. Control plane first, then data plane. Upgrade the API server, then add-ons, then nodes. The control plane may run up to two minors ahead of nodes, never behind.
  3. Read the release notes for every minor you cross. The breaking changes live there - not in a blog.
  4. Practice in dev, then prod. That is literally why this repo is modular with two envs.
  5. Back up before you start and have a rollback story. The control plane cannot be downgraded - your rollback is node/app level.
  6. Health first. A cluster that is already unhealthy (pending pods, NotReady nodes, failing add-ons) will not get healthier mid-upgrade.
Version skew policy (memorize for interviews): kube-apiserver is the reference. kubelet may be up to 3 minors older than the API server (older policy said 2); kube-proxy must be within one minor of the node's kubelet; managed add-ons (CoreDNS, VPC CNI, kube-proxy) each have their own compatibility matrix per Kubernetes version.

1 · The correct order (the runbook)

This is the sequence. Getting the order wrong is itself the most common interview failure.

  1. Pre-flight health. All nodes Ready, no long-Pending pods, all kube-system add-ons healthy, no expiring certs, enough spare capacity + IPs for a surge. kubectl get nodes,pods -A
  2. Enable audit logging (if off) so you can see deprecated API calls server-side. Let it run a few days before you trust "nothing uses removed APIs."
  3. Scan for deprecated/removed APIs with kubent and pluto against the target version - your live manifests, Helm releases, and GitOps repos.
  4. Check add-on compatibility for the target minor: aws eks describe-addon-versions --kubernetes-version 1.35 for each managed add-on; check the compatibility matrix for every Helm-installed controller.
  5. Check the data plane: AMI family (AL2 is gone at 1.33+), node OS/runtime changes, kubelet flags, custom launch templates.
  6. Clear disruption blockers: fix PDBs that allow 0 disruptions, audit fail-closed webhooks, wrap naked pods, confirm apps tolerate a rolling reschedule.
  7. Back up: Velero (or equivalent) for namespaced resources + PVs; snapshot anything stateful; export critical CRs.
  8. Upgrade the control plane one minor. aws eks update-cluster-version --name <c> --kubernetes-version 1.35 (or bump cluster_version and apply).
  9. Upgrade the managed add-ons to versions that match the new control plane - kube-proxy first / promptly, then CoreDNS and VPC CNI.
  10. Upgrade the nodes (managed node group version, or blue/green a new group and drain the old). Watch the rolling replacement respect PDBs.
  11. Validate: nodes on the new kubelet, add-ons healthy, demo app never lost quorum, workloads happy. Then repeat from step 3 for the next minor.

2 · Planted gotchas in this repo

These are wired into the Terraform and the manifests/ fixtures on purpose. Find them, fix them, understand them.

2.1 Managed add-ons lag the control plane plantedhigh

  • Who Platform/DevOps performing the upgrade.
  • What addon_versions is empty, so add-ons install at the cluster's default and do not auto-follow when you bump cluster_version. After the control-plane upgrade, VPC CNI / CoreDNS / kube-proxy sit on the old version.
  • Where modules/addons (managed add-ons), visible in kubectl get pods -n kube-system.
  • When Immediately after each control-plane bump.
  • Why EKS never upgrades add-ons for you. Lagging add-ons drift out of the supported skew window and eventually break (kube-proxy is the sharpest edge - see 2.2).
Detect
aws eks describe-addon --cluster-name <c> --addon-name kube-proxy \
  --query 'addon.addonVersion'
aws eks describe-addon-versions --addon-name kube-proxy \
  --kubernetes-version 1.35 --query 'addons[0].addonVersions[0].addonVersion'
Fix (in order, after the control plane)
# kube-proxy first, then CoreDNS, then VPC CNI
aws eks update-addon --cluster-name <c> --addon-name kube-proxy   --addon-version <v> --resolve-conflicts OVERWRITE
aws eks update-addon --cluster-name <c> --addon-name coredns      --addon-version <v>
aws eks update-addon --cluster-name <c> --addon-name vpc-cni      --addon-version <v>
# Terraform way: set addon_versions = { kube-proxy = "<v>", ... } and apply

2.2 kube-proxy version skew plantedhigh

  • Who Whoever forgets step 9 of the runbook.
  • What kube-proxy must stay within one minor of the node kubelet / control plane. Two minors behind and Services/iptables programming misbehaves.
  • Where kube-system/kube-proxy DaemonSet.
  • When If you upgrade control plane + nodes across two minors without touching kube-proxy.
  • Why It is the component people forget because "it just works" - until it doesn't.
Detect
kubectl -n kube-system get ds kube-proxy \
  -o jsonpath='{.spec.template.spec.containers[0].image}{"\n"}'
kubectl get nodes -o wide   # compare KERNEL/kubelet minor to kube-proxy
Fix
aws eks update-addon --cluster-name <c> --addon-name kube-proxy --addon-version <matching-v>

2.3 cluster-autoscaler pinned to the wrong minor plantedmed

  • Who The upgrader who bumps the cluster but not the controllers.
  • What cluster_autoscaler_image_tag = "v1.34.0". Cluster-autoscaler's image minor must track the cluster minor. After 1.34→1.35 it is a minor behind.
  • Where Helm release cluster-autoscaler in kube-system (modules/addons).
  • When Right after the control-plane upgrade.
  • Why A mismatched CA can misread node groups / new APIs and scale incorrectly. It is the canonical "controller pinned to a k8s version" gotcha (Karpenter, LB controller, metrics-server all have their own matrices too).
Detect
kubectl -n kube-system get deploy cluster-autoscaler \
  -o jsonpath='{.spec.template.spec.containers[0].image}{"\n"}'
kubectl -n kube-system logs deploy/cluster-autoscaler | grep -i version
Fix
# bump the tag to match the new cluster minor, then apply
cluster_autoscaler_image_tag = "v1.35.0"

2.4 A PodDisruptionBudget that blocks every drain plantedhigh

  • Who Anyone rolling nodes.
  • What fragile-api has 2 replicas and a PDB of minAvailable: 2 → 0 allowed disruptions → kubectl drain and the managed node group rolling upgrade stall on this namespace until they time out.
  • Where manifests/20-blocking-pdb.yaml, namespace upgrade-gauntlet.
  • When During node rotation (step 10).
  • Why The single most common reason a node upgrade "hangs." PDBs are correct in spirit and lethal when minAvailable == replicas.
Detect
kubectl get pdb -A          # look for ALLOWED DISRUPTIONS = 0
kubectl get pdb -A -o json | jq -r '.items[]
  | select(.status.disruptionsAllowed==0)
  | "\(.metadata.namespace)/\(.metadata.name)"'
Fix
# give it headroom: raise replicas, or loosen the budget
kubectl -n upgrade-gauntlet scale deploy/fragile-api --replicas=3
# or set minAvailable: 1  (or maxUnavailable: 1)

2.5 A fail-closed admission webhook plantedhigh

  • Who The whole cluster, if this scope were broad.
  • What A ValidatingWebhookConfiguration with failurePolicy: Fail whose backing Service has no healthy endpoints. It is scoped inert here (matches a label nothing carries) - but that is exactly the shape that bricks a cluster when the backend is evicted during a drain.
  • Where manifests/40-failing-webhook.yaml.
  • When During node rotation, if a real Fail webhook's pods land on the drained node and don't reschedule (deadlock: you can't create pods to recover).
  • Why "Webhook took down the cluster during maintenance" is a classic postmortem. Audit failurePolicy, timeout, scope, and backend HA before upgrading.
Detect
kubectl get validatingwebhookconfigurations,mutatingwebhookconfigurations -o json \
  | jq -r '.items[] | .metadata.name as $n | .webhooks[]?
    | select(.failurePolicy=="Fail")
    | "\($n)  timeout=\(.timeoutSeconds)  \(.clientConfig.service.namespace)/\(.clientConfig.service.name)"'
Fix
# make the backend HA + anti-affinity, cap timeoutSeconds, scope tightly,
# and for non-critical webhooks prefer failurePolicy: Ignore.
kubectl delete validatingwebhookconfiguration gauntlet-fail-closed-demo

2.6 Non-canonical CIDR (1.36 strict validation) plantedmed

  • Who Anyone editing this object on 1.36.
  • What A NetworkPolicy with ipBlock.cidr: 10.0.5.7/16 - host bits set. Fine on 1.34/1.35. On 1.36 StrictIPCIDRValidation rejects any create/update carrying a non-canonical CIDR (or leading zeros). Existing objects are ratcheted, so it lurks until the next edit.
  • Where manifests/30-noncanonical-cidr-netpol.yaml.
  • When First edit/reapply after reaching 1.36 (very GitOps-unfriendly).
  • Why A real 1.36 release-note action item that is invisible until it bites.
Detect
# audit manifests/GitOps for CIDRs with host bits or leading zeros
grep -rEn '([0-9]{1,3}\.){3}[0-9]{1,3}/[0-9]+' manifests/ | grep -Ev '\.0/'
Fix
cidr: 10.0.0.0/16   # canonical network address

2.7 A naked singleton pod plantedlow

  • Who The "service" that quietly disappears.
  • What legacy-singleton is a bare Pod with no controller. Node rotation evicts it and it never comes back.
  • Where manifests/50-singleton-pod.yaml.
  • When The moment its node is drained.
  • Why Only controller-managed pods reschedule. Naked pods = guaranteed downtime on any node lifecycle event.
Detect
kubectl get pods -A -o json | jq -r '.items[]
  | select(.metadata.ownerReferences==null)
  | "\(.metadata.namespace)/\(.metadata.name)"'
Fix
# wrap it in a Deployment (replicas + rescheduling)
kubectl -n upgrade-gauntlet create deployment legacy --image=nginx:1.27-alpine --replicas=2

2.8 aws-auth vs access entries plantedmed

  • Who Anyone managing cluster access / migrating auth.
  • What authentication_mode = "API_AND_CONFIG_MAP" - both the legacy aws-auth ConfigMap and the new access entries are active. Confusing to reason about; easy to lock yourself out mid-migration.
  • Where modules/eks access config; kube-system/aws-auth and EKS access entries.
  • When During an access-management migration (increasingly bundled with upgrades).
  • Why Access entries are the modern, IAM-native way and remove a whole class of aws-auth footguns. Know how to move and how not to lose admin.
Detect
kubectl -n kube-system get configmap aws-auth -o yaml
aws eks list-access-entries --cluster-name <c>
aws eks describe-cluster --name <c> --query 'cluster.accessConfig.authenticationMode'
Fix
# create access entries for every principal that needs access, verify,
# THEN move authentication_mode to API and retire aws-auth.
aws eks create-access-entry --cluster-name <c> --principal-arn <arn>
aws eks associate-access-policy --cluster-name <c> --principal-arn <arn> \
  --policy-arn arn:aws:eks::aws:cluster-access-policy/AmazonEKSClusterAdminPolicy \
  --access-scope type=cluster

2.9 Small subnets → IP exhaustion on surge plantedmed

  • Who The upgrade that needs to launch surge nodes and pods.
  • What Deliberately small subnets (/24). The VPC CNI assigns real VPC IPs to every pod; a rolling upgrade briefly doubles nodes+pods and can exhaust the subnet.
  • Where modules/vpc subnet CIDRs.
  • When During surge (new nodes come up before old drain).
  • Why "Pods stuck in ContainerCreating - failed to assign an IP" mid-upgrade is a real outage. Plan IP space and/or enable prefix delegation.
Detect
aws ec2 describe-subnets --filters Name=tag:kubernetes.io/role/internal-elb,Values=1 \
  --query 'Subnets[].[SubnetId,AvailableIpAddressCount]' --output table
kubectl get pods -A | grep -E 'ContainerCreating|FailedCreatePodSandBox'
Fix
# enable prefix delegation (more pods per ENI, denser IP use)
kubectl -n kube-system set env ds/aws-node ENABLE_PREFIX_DELEGATION=true
# or widen private_subnet_cidrs / add subnets and reapply

2.10 Audit logging is off plantedlow

  • Who The person trying to prove nothing uses a deprecated API.
  • What enabled_cluster_log_types = []. Cheap by default, but you are blind to which clients still call deprecated/removed APIs.
  • Where modules/eks logging.
  • When Before any upgrade that crosses an API removal.
  • Why kubent/pluto scan manifests; the audit log catches the live callers (that controller you forgot about).
Detect / Fix
# turn on audit (+ api) logging BEFORE the upgrade, let it soak
enabled_cluster_log_types = ["audit", "api"]
# then query the apiserver_requested_deprecated_apis metric / CloudWatch Logs Insights

2.11 Public API endpoint open to the world plantedmed

  • Who Security review.
  • What public_access_cidrs = ["0.0.0.0/0"]. Convenient for a test, wrong for prod.
  • Where modules/eks vpc_config.
  • When Any hardening pass; especially prod.
  • Why The API server should be reachable only from known networks (or private-only + a bastion/VPN).
Detect / Fix
aws eks describe-cluster --name <c> --query 'cluster.resourcesVpcConfig.publicAccessCidrs'
# restrict:
public_access_cidrs = ["203.0.113.0/24"]   # your office/VPN
# or endpoint_public_access = false (private-only)

2.12 Node OS & runtime changes (AL2023 · cgroup v1 · containerd 2.0) plantedhigh

  • Who Anyone with custom AMIs, bootstrap scripts, or cgroup-v1 assumptions.
  • What Nodes run AL2023 (AL2 AMIs stop at 1.33). Across the path: 1.35 removes cgroup v1 (kubelet refuses to start on a v1 node) and is the last containerd 1.x release; 1.36 requires containerd 2.0.
  • Where ami_type = "AL2023_x86_64_STANDARD" in modules/eks; anything with a custom launch template.
  • When The 1.35 and 1.36 hops.
  • Why Custom AMIs / user-data that assume AL2, cgroup v1, containerd 1.x, or the removed --pod-infra-container-image kubelet flag will fail to boot or join.
Detect
kubectl get nodes -o wide     # OS-IMAGE, CONTAINER-RUNTIME, KERNEL
kubectl get nodes -o jsonpath='{range .items[*]}{.status.nodeInfo.containerRuntimeVersion}{"\n"}{end}'
Fix
# use EKS-optimized AL2023 (default here), remove AL2-only bootstrap assumptions,
# drop the removed --pod-infra-container-image flag, verify cgroup v2 + containerd 2.x
# before the 1.35 / 1.36 hops. Roll nodes AFTER the control plane.

3 · Version-boundary specifics (1.34 → 1.36)

The public AWS/Kubernetes release-note headlines for each hop. This is what "read the release notes" actually turns up.

HopWhat changes (and what to do)
→ 1.34No EKS-optimized AL2 AMI (migrate to AL2023). AppArmor deprecated. VolumeAttributesClass GA - self-managed EBS CSI sidecars may need pinning on pre-1.34. DRA core APIs GA. Endpoints API deprecated (use EndpointSlices).
→ 1.35cgroup v1 removed - kubelet refuses to start on cgroup-v1 nodes. Last containerd 1.x release - move to containerd 2.0 before the next hop. IPVS kube-proxy mode deprecated. --pod-infra-container-image kubelet flag removed - strip it from custom AMIs.
→ 1.36containerd 2.0 required. StrictIPCIDRValidation on by default - non-canonical CIDRs / leading zeros rejected on create/update. gitRepo volume removed (kubelet refuses). IPVS mode removed. Service.spec.externalIPs deprecated.
⚠️ 1.33 is excluded on purpose. Its EKS standard support ends 2026-07-29; past that the control plane jumps to extended-support pricing (~6×). We start at 1.34 to stay cheap. Falling behind into extended support is itself the lesson.

5 · General pre-upgrade checklist

The full sweep, beyond what's planted here.

  • Cluster health: nodes Ready, no stuck Pending/CrashLoop, control plane + add-ons healthy.
  • Deprecated/removed APIs: kubent + pluto on manifests, Helm, GitOps; audit logs for live callers.
  • Add-on compatibility: every managed add-on and Helm controller against the target minor.
  • Node plan: AMI family (AL2023), OS/runtime (cgroup v2, containerd 2.0), custom launch templates, kubelet flags.
  • Capacity & surge: spare vCPU quota, spare IPs, max_unavailable/surge settings; enough room to drain one node.
  • PDBs: none at 0 allowed disruptions; CoreDNS/critical add-ons have sane PDBs.
  • Admission webhooks: failurePolicy, timeout, scope, backend HA - fail-open the non-critical ones.
  • Naked pods & singletons: everything important is controller-managed; StatefulSets tolerate reschedule.
  • Stateful/EBS: EBS is AZ-locked - a pod must reschedule into its volume's AZ; check topology + PVs.
  • Certs & webhooks: cert-manager/webhook certs not expiring during the window.
  • Identity: OIDC provider intact; IRSA/Pod-Identity roles valid; access entries / aws-auth reconciled.
  • Service quotas: EIP, ENI, instance vCPU, NAT - a surge can hit a limit.
  • Backups: Velero + snapshots + critical CR exports; know your rollback (control plane can't downgrade).
  • Terraform hygiene: upgrading a shared module can force replacements - plan and read every replace; bump cluster_version deliberately, one minor per apply.
  • Probes & graceful shutdown: readiness/liveness sane, terminationGracePeriodSeconds honored, connection draining.
  • Dev first: do the whole thing in dev, then prod.

kubent                                   # kube-no-trouble, scans live cluster vs target
pluto detect-files -d ./manifests        # scan files
pluto detect-helm -owide                 # scan helm releases

## EKS add-on version matrix

## what's actually installed / health
kubectl get nodes -o wide
kubectl get pods -A -o wide
kubectl get pdb,validatingwebhookconfigurations,mutatingwebhookconfigurations -A
kubectl get pods -A -o json | jq -r '.items[] | select(.metadata.ownerReferences==null)
  | "\(.metadata.namespace)/\(.metadata.name)"'   # naked pods

## upgrade the control plane / nodes
aws eks update-cluster-version --name  --kubernetes-version 1.35
aws eks update-nodegroup-version --cluster-name  --nodegroup-name

cluster_version = "1.35"        # then apply -> upgrades control plane
node_version    = "1.35"        # then apply -> rolls nodes
addon_versions  = { kube-proxy = "", coredns = "", vpc-cni = "" }

7 · Why these gotchas were chosen

Every planted item maps to a real category that shows up in production upgrades and interviews:

  • Add-on skew, kube-proxy, cluster-autoscaler - the "you upgraded the control plane but forgot the data plane / controllers" family. The most common real mistake.
  • Blocking PDB, fail-closed webhook, naked singleton - the disruption-budget / admission / workload-shape traps that turn a routine node rotation into an outage.
  • Non-canonical CIDR, node OS/runtime - version-boundary behavioral changes (1.35/1.36) that kubent will not catch, so you learn to read release notes, not just run a scanner.
  • aws-auth vs access entries, open endpoint, audit off - the identity/access/security items bundled into modern upgrades.
  • Small subnets - the capacity/IP-space planning that surge upgrades expose.

They are cheap to run (software-layer, on 3 small nodes), safe (nothing bricks the cluster by default), and detectable with standard tooling - so the drill is about process and knowledge, not luck.

8 · Cost & teardown

  • Control plane: $0.10/hr standard (1.34-1.36) · $0.60/hr extended support (why we avoid 1.33 after 2026-07-29).
  • 3× t3.medium SPOT ≈ $0.04/hr · single NAT ≈ $0.045/hr · a promptly-destroyed session ≈ $1-2.
  • Apply one env at a time - dev + prod together doubles the control-plane cost.
🔻 make down ENV=dev when you finish. The only expensive mistakes are forgetting to destroy and drifting onto extended support.
